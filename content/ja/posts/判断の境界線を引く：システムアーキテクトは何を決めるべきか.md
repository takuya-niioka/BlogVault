## 「自分は何を決めるべきか？」という終わらない迷い

「次世代制御機能の仕様変更、自分が決めていいのか？それとも上司に相談すべきか？」

上司に相談すると「自分で考えて」と言われ、自分で決めて報告すると「事前に相談してほしかった」と言われる。

「この判断、後輩に任せていいのか？」

任せると方向性が違ってしまい、細かく指示すると「マイクロマネジメント」と言われる。後輩は指示待ちになり、自分の負担は減らない。

**どっちに転んでも詰む。**

これが、判断範囲が曖昧なことの苦しみです。システムアーキテクトやテックリードなら、一度は経験したことがあるのではないでしょうか。

### なぜ判断範囲が曖昧になるのか

判断範囲が曖昧になる理由は4つあります。**明示的に定義されていない**（「空気を読んで判断」という暗黙知に依存）、**ケースバイケースで判断している**（一貫性のある基準がない）、**自分の役割が不明確**（肩書きはあっても期待が曖昧）、**工数不足という現実**（すべて自分で判断するのは物理的に不可能）です。

### 判断範囲が曖昧だと何が起きるか

判断範囲が曖昧なまま仕事を続けると、悪循環が始まります。

**あなた自身**は細かい判断に認知資源を費やし疲弊し、本来やるべき戦略的判断に時間を使えません。上司からは「積極性がない」と評価され、キャリアが停滞します。**後輩**は何を期待されているのか分からず、指示待ちになり、成長の機会を失います。**上司**は「自分で考えてほしい」と思っているが何を報告してほしいのか伝わらず、チーム全体のパフォーマンスが上がりません。

この悪循環を断ち切るには、**判断範囲を明確にすること**が必要です。


## 判断の階層構造を理解する

判断範囲を明確にする第一歩は、判断には階層があることを理解することです。

### 5つの判断階層

組織における意思決定は、大きく5つの階層に分けられます。なぜ5階層なのか？それは、**時間軸・不確実性・影響範囲**という3つの次元で判断が明確に分離できるからです。多すぎると複雑になり、少なすぎると粒度が粗くなります。5階層は、実務上最も機能する分割です。

| 階層              | 役割                       | 判断内容                             | 時間軸     | 不確実性           | 影響範囲             | V字プロセス位置                                   | 判断の例                                                                             |
| --------------- | ------------------------ | -------------------------------- | ------- | -------------- | ---------------- | ------------------------------------------ | -------------------------------------------------------------------------------- |
| **Layer 5**     | 経営層                      | 事業方針、予算配分、組織体制の決定                | 数年      | ★★★★★<br>非常に高い | 会社全体             | 事業戦略                                       | Level 3自動運転に5年で100億円投資                                                           |
| **Layer 4**     | 部門長<br>チーフエンジニア          | システム全体の方向性、アーキテクチャの決定            | 数ヶ月〜1年  | ★★★★☆<br>高い    | 部門全体<br>複数機能横断   | **左側最上位**<br>（要求分析・システム要求）                 | カメラ中心 vs LiDAR中心アーキテクチャ<br>システム要求仕様の策定                                           |
| **Layer 3**<br> | システムアーキテクト<br>システムズエンジニア | 機能要件、性能目標、技術選択、Validation計画の決定   | 数週間〜数ヶ月 | ★★★☆☆<br>中     | 機能チーム<br>担当システム  | **左側**（要求・要件定義）<br>**右側**（Validation）      | 【要件】機能要件の定義、性能目標の設定<br>【戦略】AEB夜間性能70%→82%向上のためレーダー採用<br>【Validation】検証方法の定義、計画策定 |
| **Layer 2**     | シニアエンジニア                 | 詳細設計、実装アーキテクチャ、Verification戦略の決定 | 数日〜数週間  | ★★☆☆☆<br>低〜中   | モジュール<br>コンポーネント | **左側**（詳細設計）<br>**右側**（Verification・統合テスト） | センサーフュージョンアルゴリズムの設計<br>統合テスト戦略、カバレッジ目標の設定                                        |
| **Layer 1**     | エンジニア                    | コーディング、単体テスト、バグ修正の判断             | 数時間〜数日  | ★☆☆☆☆<br>低     | ファイル<br>関数レベル    | **最下層**<br>（実装とテスト）                        | 関数の実装方法、データ構造の選択<br>単体テストケース設計、デバッグ手法                                            |

**Layer 3における2つの視点**[システムアーキテクトの本質：正しいものを作るということ](/ja/posts/what-systems-architects-actually-do/)：
- **Systems Engineer視点**：「このシステムは正しく機能するか？」（要件定義・Validation）
- **System Architect視点**：「このシステムは正しいものか？」（戦略・価値定義）

**後輩がLayer 3にいる場合**：機能要件の一部を後輩に任せ、方向性を50%時点で確認します。

#### もう一つの軸：プロジェクト管理（PL/PM）との役割分担

上記のLayer 1-5は**技術的な判断の階層**です。これとは別に、**プロジェクト管理軸**が存在します。特に車両機種チームでは、Project Leader（PL）やProject Manager（PM）が存在し、システムアーキテクトとは異なる役割を担います。

**重要な前提：システムアーキテクトはPM業務を「しない」**

システムアーキテクト（Layer 3）とPL/PMは、同じ判断事項に対して異なる視点から関与しますが、最終的な責任範囲は明確に異なります。

**システムアーキテクト vs PL/PMの役割分担：**

| 判断事項 | システムアーキテクト（技術軸） | PL/PM（プロジェクト管理軸） |
|---------|--------------------------|------------------------|
| **What（何を作るか）** | 技術仕様・性能目標を**提案** | スコープ・優先順位を**決定** |
| **How（どう作るか）** | 技術選択・アーキテクチャを**決定** | リソース配分・タスク分解を**決定** |
| **When（いつまでに）** | 技術的実現性を**評価** | スケジュール・マイルストーンを**設定** |
| **Who（誰が）** | 技術スキル要件を**定義** | 人員アサイン・チーム編成を**決定** |
| **リスク** | 技術リスクを**評価・対策提案** | プロジェクトリスク全体を**管理** |
| **最終責任** | **技術的成立性** | **プロジェクト成功（QCD）** |

**判断の流れ（例：4Dレーダー採用の場合）：**

1. **システムアーキテクト**が技術検討し、「性能要求を満たせる」と判断
2. **PL/PM**がスケジュール・予算・他機能への影響を評価
3. 両者が合意して初めて「採用」が決定される
4. 採用後、**システムアーキテクト**は技術実現を担保し、**PL/PM**はプロジェクト全体を推進

**境界を曖昧にすると起きる問題：**
- システムアーキテクトがPM業務に埋没 → Layer 3の戦略判断に時間を使えない
- 技術とスケジュールの責任が混在 → 「技術的には最高だが納期に間に合わない」
- 意思決定の遅延 → 誰が最終判断するのか不明確

**システムアーキテクトの役割は、技術的な判断に集中し、PL/PMに対して技術的インプットを提供すること**です。プロジェクト管理はPL/PMに任せます。

### 自分の階層を見極める

自分がどの階層で主に判断すべきかを見極めることで、何が得られるのでしょうか？

#### 階層を明確にすることで得られる3つのメリット

| メリット       | 階層が曖昧な状態                                        | 階層を明確にした状態                                            |
| ---------- | ----------------------------------------------- | ----------------------------------------------------- |
| **① 認知負荷** | ✗ すべての判断に全力<br>✗ 脳が疲弊<br>✗ 常に迷う                 | ✓ Layer 1-2は後輩に委譲<br>✓ Layer 3に認知資源を集中<br>✓ 迷いが消える    |
| **② 上司評価** | ✗ Layer 1-2に時間を費やす<br>✗ 「積極性がない」と評価<br>✗ キャリア停滞 | ✓ Layer 3で主体的に判断<br>✓ Layer 4（上司）に提案<br>✓ 「積極的」と評価される |
| **③ 後輩育成** | ✗ 何を任せていいか不明<br>✗ 指示待ち状態<br>✗ 成長しない             | ✓ Layer 1-2の判断を任せる<br>✓ 成功体験を積む<br>✓ 自分の負担も軽くなる       |

#### 自分の階層を見極める4つの判断基準

では、自分がどの階層で主に判断すべきか？それを見極める判断基準を表にまとめます：

| 判断基準         | Layer 5<br>（経営層） | Layer 4<br>（部門長） | Layer 3<br>（システムアーキテクト）      | Layer 2<br>（シニアEng.） | Layer 1<br>（エンジニア） |
| ------------ | ---------------- | ---------------- | -------------------------------- | -------------------- | ------------------ |
| **役職・経験**    | 経営層<br>取締役       | 部門長<br>チーフエンジニア  | 経験5-10年<br>システムアーキテクト        | 経験3-7年<br>シニアエンジニア   | 経験1-3年<br>若手エンジニア  |
| **組織での呼称**   | CEO, CTO         | 部長, GL           | 「システムアーキテクト」<br>「主担」         | 「テックリード」<br>「担当」     | 「メンバー」<br>「担当者」    |
| **意思決定の時間軸** | 数年               | 数ヶ月〜1年           | 数週間〜数ヶ月                      | 数日〜数週間               | 数時間〜数日             |
| **影響範囲**     | 会社全体             | 部門・複数機能          | 機能チーム・担当システム                 | モジュール・コンポーネント        | ファイル・関数            |
| **V字プロセス**   | 事業戦略             | 要求分析             | 要件定義<br>Validation           | 詳細設計<br>Verification | 実装<br>単体テスト        |
| **判断の例**     | 事業投資判断           | システム要求仕様         | 機能要件<br>Validation計画<br>技術選択 | 詳細設計<br>統合テスト戦略      | 実装方法<br>単体テスト      |

#### 階層を見極めないとどうなるか？

逆に、自分の階層を見極めないままだと：
- Layer 1-2の細かい判断に時間を取られる（50%以上）
- Layer 3の戦略的判断に使える時間が減る（30%以下）
- 上司からは「積極性がない」と評価される
- 後輩は「何を判断していいのか分からない」と迷う
- 結果として、自分も後輩も成長しない

**自分がどの階層を主戦場とすべきかを見極めることが、判断範囲を明確にする第一歩です。**

## 判断範囲を戦略的に設計する

自分のレイヤーが分かったら、次は具体的に判断範囲を設計します。ここでは3つの境界線を引く必要があります：上司との境界、自分の範囲、後輩との境界です。

### 上のレイヤー（上司）との境界線

まず、上司との境界線を明確にしましょう。何を上司に判断してもらい、何を自分で決めるのか。

#### 上司に判断を仰ぐべき4つのシグナル

以下のシグナルが1つでも当てはまる場合は、上司の判断領域です：

| シグナル       | 判断基準                     | 具体例                              |
| ---------- | ------------------------ | -------------------------------- |
| **予算影響**   | 100万円以上の投資<br>（会社により異なる） | 新しいセンサー導入に3000万円<br>→ 予算権限は上司にある |
| **部門横断調整** | 他部門との交渉・協議が必要            | 購買部門との交渉<br>品質部門とのISO 26262協議    |
| **会社方針整合** | 事業戦略レベルの判断               | Level 3自動運転への投資判断<br>競合との差別化戦略   |
| **高リスク**   | 失敗したら大問題<br>説明責任が重い      | 安全性に関わる根本的設計変更<br>規制対応できないリスク    |

**判断フロー：**

```
判断事項が発生
   ↓
[4つのシグナルチェック]
   ↓
YES（1つでも該当）─→ 上司に判断を仰ぐ
   ↓
NO（すべて該当しない）─→ 自分で判断（事前通知）
```

#### 上司への上げ方：判断しやすい形で提案する

上司に判断を仰ぐ時、「どうしましょうか？」では丸投げです。技術的な選択肢を整理し、推奨案を明示し、理由とトレードオフを説明する。これが上司の判断を助けます。

「次世代センサーについて、3つの選択肢があります。私はOption B（4Dレーダー追加、3000万円）を推奨します。理由は規制要求（夜間検出率80%）をクリアでき（82%達成見込み）、予算内（3500万円以内）で18ヶ月で実現可能だからです。Option Aは低コストですが規制未達のリスクがあり、Option Cは高性能ですが予算超過します。最終判断をお願いします」

このように、**選択肢・推奨案・理由・トレードオフ・リスク**の5要素を揃えれば、上司は判断しやすくなります。

#### 自分で決める時の伝え方：事前通知で自律性を示す

一方、自分の判断範囲内の決定は、「事前通知」の形を取ります。

「次世代センサーのアルゴリズム選定について、Option B（Kalman Filter改良版）で進めます。理由は性能要求を満たし実装リスクも低く、開発費200万円で私の裁量範囲内だからです。問題があれば◯月◯日までにご指摘ください」

事前通知（サプライズを避ける）、理由の明示、異議申し立て期間の設定。これで自律性を示しつつリスクを回避し、上司からは「積極性がある」と評価されます。

### 下のレイヤー（後輩）との境界線

次に、後輩との境界線を明確にします。何を後輩に任せ、何を自分でやるのか。

#### 後輩に任せるべき4つのシグナル

以下のシグナルがすべて当てはまる場合は、後輩に任せるべきです：

| シグナル        | 判断基準                    | 具体例                      | 狙い        |
| ----------- | ----------------------- | ------------------------ | --------- |
| **実装レベル**   | Layer 1-2の判断            | アルゴリズム詳細実装<br>テストコード設計   | 適切な階層分離   |
| **リカバリー可能** | 失敗しても大問題にならない           | プロトタイプ試作<br>内部モジュールリファクタ | 学習機会の提供   |
| **スキル内**    | 後輩が理解できる範囲<br>過去に似た経験あり | 使ったことある技術<br>類似課題の経験     | 成功体験の積み上げ |
| **時間余裕**    | 試行錯誤の時間を確保できる           | 急ぎでない改善タスク<br>リファクタリング   | じっくり学べる環境 |

**委譲判断フロー：**

```
判断事項（Layer 1-2レベル）が発生
   ↓
[4つのシグナルチェック]
   ↓
すべてYES ─→ 後輩に委譲（Why・制約・判断基準を明示）
   ↓
1つでもNO ─→ 自分で判断 or 一緒に判断
```

#### 後輩への任せ方：境界を明示する

後輩に仕事を任せる時、「このモジュール、よろしく」では丸投げです。後輩は何をどこまでやればいいのか分かりません。

「センサーフュージョンモジュールの実装をお願いします。目的はカメラとレーダーのデータを統合し物体検出精度を向上させることです。制約は処理時間50ms以内、メモリ100MB以内、追加部品なし。判断基準は検出精度90%以上、誤検出率1%以下です。方法は任せます。◯月◯日までに設計案を持ってきてください。進捗は50%時点で確認します」

**Why（目的）・制約・判断基準**を伝え、**How（方法）**は任せ、中間報告のタイミングを設定する。これで後輩は「自分で判断していい範囲」と「守るべき制約」が明確になります。

#### 任せた後のフォロー：方向性だけ確認する

中間報告（50%時点）では、方向性だけを確認します。「どのアプローチを選びましたか？なぜそれを選びましたか？」と問い、細かい実装指示はしません。細かく指示すると後輩の学習機会を奪います。方向性が正しければ、Howは任せます。

最終報告では、「なぜこの判断をしたのか、説明してください」と問います。後輩は自分の判断を言語化する訓練になり、Decision Recordとして記録すれば組織の学習資産になります。

### もしあなたがシステムアーキテクトなら：レイヤー（Layer 3）に集中する

上司との境界、後輩との境界を明確にしたら、Layer 3（機能戦略）に集中します。

**Layer 3で判断すべきこと（システム屋の視点）：**
- 機能要件の定義と優先順位付け
- Validation計画の策定（どう検証するか）
- 性能とコストのトレードオフ
- 技術選択（センサー、アルゴリズム）
- リスクと品質目標の設定

**後輩がLayer 3にいる場合：**
- 後輩に機能要件の一部を任せる（Why・制約・判断基準を明示）
- 方向性を50%時点で確認（細かい指示はしない）
- Validation計画は一緒に議論し、後輩の判断を育てる

**理想的な時間配分：**
- Layer 4への関与：20%（上司への提案、システム要求への意見）
- **Layer 3（自分）：60%（機能要件、Validation計画、技術判断）** ← ここに集中
- Layer 2への支援：15%（詳細設計の方向性確認）
- Layer 1への支援：5%（緊急時のみ）

## 判断範囲を明確にするツール

判断範囲を明確にするために、実務で使える2つのツールを紹介します。

### Decision Rights Matrix（判断権限マトリクス）

Decision Rights Matrix[^6]は、「誰が何をするか」を明確にするツールです。RASICモデル[^5]という枠組みを使います。

**RASICモデル：**
- **R = Responsible（実行責任）**：実際に作業を行う人
- **A = Accountable（説明責任）**：最終判断・承認をする人（**各項目に1人だけ**）
- **S = Supportive（支援）**：作業を支援する人
- **C = Consulted（相談される）**：意見を求められる人
- **I = Informed（情報共有される）**：結果を知らされる人

#### 実例：次世代センサー選定のRASICマトリクス

| 項目 | 上司<br>（部門長） | 自分<br>（アーキテクト） | 後輩<br>（エンジニア） | 品質部門 | 購買部門 | PL/PM |
|------|---------|---------|---------|----------|----------|-------|
| **技術調査** | I | **R/A** | S | - | - | I |
| **選択肢整理** | C | **R/A** | S | - | - | C |
| **コスト見積** | C | **R** | S | - | C | **A** |
| **最終決定** | **A** | **R** | I | C | I | C |
| **詳細設計** | I | C | **R/A** | S | - | I |
| **実装** | I | C | **R/A** | - | - | I |

**このマトリクスのポイント：**

✓ 誰が何をするか一目瞭然
✓ 「これは誰の判断?」という迷いがなくなる
✓ ステークホルダー全員が理解できる
✓ **Accountable（A）は各項目で1人だけ** ← 最重要原則

複数の人がAになると、「結局誰が決めるの?」という混乱が生じます。Aは必ず1人に絞りましょう。

### Decision Record（判断の記録）

重要な判断は記録に残しましょう。Decision Recordとして以下を記録します：

**記録する内容：**
- 判断事項（何を決めたか）
- 判断レイヤー（Layer 3など）
- 理由とトレードオフ
- 前提条件
- 誰に報告し、誰に委譲したか

Decision Recordは、後で振り返る時や前提条件が変わった時の判断見直し[^1]に役立ちます。完璧である必要はありません。まずは簡単なフォーマットで始めて、チームで改善していけばいいのです。

## 判断範囲を明確にすることは、中堅がやるべき最初の仕事

判断範囲を明確にすることは、優先順位付けや価値定義の前提条件です[^2]。

なぜなら、**判断範囲が曖昧だと、そもそも何を優先すべきか分からない**からです。

### 判断範囲の設計は戦略的行為

判断範囲を設計することは、単なる業務分担ではありません。これは戦略的な行為です。

**上のレイヤー（上司）との境界：**
✓ 何を上げ、何を自分で決めるか
✓ 4つのシグナル（予算、部門調整、会社方針、リスク）で判断
✓ 事前通知で自律性を示す
→ 「積極性がある」として評価される

**自分のレイヤー（Layer 3）：**
✓ 機能戦略に集中する
✓ 質の高い判断に時間を使う
✓ 戦略的思考を磨く[^3]

**下のレイヤー（後輩）との境界：**
✓ 何を任せるか
✓ 4つのシグナル（実装レベル、リカバリー可能、スキル内、時間余裕）で判断
✓ Why・制約・判断基準を明示し、Howは任せる
→ 後輩が成長する

### 使えるツール

**Decision Rights Matrix（RASIC）：**
誰が何をするかを明確にする。Accountableは1人だけ。

**Decision Record（判断記録）：**
判断を記録し、振り返る。後輩が学べる。上司に説明できる。

### 境界は固定ではない

判断範囲は、一度決めたら終わりではありません。定期的に見直す必要があります。

**見直しのタイミング：**
✓ 後輩が成長したら、任せる範囲を広げる
✓ 環境が変わったら、境界も変わる
✓ 3ヶ月ごとに振り返る

**柔軟に調整する：**
✓ 完璧な境界は存在しない
✓ 試行錯誤しながら最適化する[^4]
✓ チームで合意しながら進める

### 最後のメッセージ

判断範囲が明確になると：
✓ 自分は本来の仕事に集中できる
✓ 後輩は成長する
✓ 上司からは評価される
✓ チーム全体のパフォーマンスが上がる

そして、次の問いが立ちます：

**「では、自分の判断範囲（Layer 3）の中で、やるべきことが10個ある時、何を優先するのか？」**

これについては、次回の「戦略的優先順位付けの技法」で詳しく解説します。

---

**判断の境界線を引くことは、システムアーキテクトの最初の仕事です。**

まずは、自分のレイヤーを見極めることから始めましょう。そして、Decision Rights Matrixを作り、上司と後輩に共有しましょう。完璧である必要はありません。まずは一歩を踏み出すことが重要です。

そして、判断を記録しましょう。Decision Recordは、あなたの思考を整理し、チームの学習資産になります。
